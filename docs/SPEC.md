# Chiaki AI 仕様書

2026-07-10 時点の正式仕様。Lipple 業務を観測する Slack 常駐 AI「Chiaki AI」（Hermes management プロファイル）の全体像。運用手順は CLAUDE.md、Codex への指示書は AGENTS.md。

## 1. 設計原則

- **口は GPT・手は決定論**。応対の判断と文面は GPT 5.5 が会話の文脈から決める。実行（Notion 起票・投稿編集・Codex 起動・デプロイ）は定義済みのアクションに限定され、すべて決定論のゲート（権限・確認・冪等性・上限）を通る。
- 任意コード実行を持つ会話エージェント（gateway / execute_code）は**封印**。復活させない。
- LLM の経路: 中枢 = GPT 5.5（ChatGPT サブスク・Codex と認証共用）。不通時は Haiku 4.5 に自動退避し #8902 に日次で控え。LLM が文面を作った発言には末尾にモデル表記（例:（GPT 5.5））。タグ無し＝固定文/決定論。
- GPT の出力が壊れた場合は従来の決定論ロジックに自動フォールバック＝無反応にならない。

## 2. 会話系（窓口）

起点は戸田さんの発話。listener（Socket Mode 常駐）が即時に拾い、cron がバックストップ。

### 2.1 受付

- `@Chiaki AI 〜`（どのチャンネルでも）／#8902・#5902 の戸田さんの投稿・スレッド返信。
- 権限: コード変更指示・定型業務化・Codex 起動を受け付けるのは**戸田さんの user ID（U9R35H06L）のみ**。他メンバーの @Chiaki AI は受領＋戸田さんへの引き継ぎのみ（トップレベルの新規メンション・直近1時間限定）。

### 2.2 会話コア（Phase A+B+C・2026-07-10）

すべての応対（初回・確認ターン・Codexスレッド）は共通の会話コア（lib/convo）を通る。話す前に必ず読む文脈パッケージ＝①スレッド全文（自分の発言を「あなた」と明示） ②このスレッドでの自分の状態・行動の事実（裁定の状態・出した依頼文・完了通知・起票状態とURL・Codex作業とブランチ） ③修正報告の知識 ④システム状態（Codex利用上限中など） ⑤長期記憶と直近の別スレッドのやりとり（Phase C）。アクションは統一集合（propose/file/revise/cancel/edit_post/company_rule/codex_continue/deploy_request/retract/answer/silent）をモード別の許可リストで検証。出力不正・LLM不通は従来ロジックへ自動フォールバック。

**スレッドを跨ぐ記憶（Phase C・2026-07-10）**: 会話コアの判断が実行に採用された時だけ、会話台帳（convo_memory.json・直近120件）に記録される（フォールバックした判断は残らない＝記憶と事実を食い違わせない）。夜間の convo-memory（平日21:10）が台帳から「決定・好み・注意点」を短文の長期記憶（最大25件・各120字）に蒸留する（判断はGPT・検証と保存は決定論。出力不正・全消し提案は旧記憶を温存。新しい会話が無い日はLLM非起動）。会話コアは毎回この長期記憶と直近の別スレッドのやりとりを注入＝「さっきの件」「昨日の話」がスレッドを跨いで通じる。

初回応対の分類方針:

- 「なぜ」「どうして」を含む発話は、不満が同居していても question 優先＝まず答える。
- Chiaki AI の投稿の表記・整形の指摘（改行・空行・記号）は issue でなく edit＝その場で投稿を直す。issue にするのはコードの動作の再現する不具合だけ。
- 起票は必ず一度きいてから（2ターン）。未対応/未承認は本番不反映＝承認は人。

### 2.3 確認ターン

提案後の戸田さんの返信も会話コアが判断（提示文・返事はすべて文脈から生成＝「登録してもいいですか？」等の紋切り廃止）:

- **file**: 承認（「OK」「Issueに追加で」等。質問や依頼が同居していても承認部分を消費し、返事の中で同居した話にも応じる。新しい指摘が同居していれば起票後に次の確認へ）
- **revise**: 内容・振り分けの変更（「それRuleね」）→ 修正案を見せて確認
- **cancel**: 却下
- **edit_post**: 投稿の手直し
- **answer_only**: 質問・雑談＝答えるだけ（提案は開いたまま。案内の定型文は出さない）

### 2.4 知識源

- **修正報告の記録**（#8902 の「報告：」投稿・直近6件）を回答の知識として参照＝「なぜ？」に「こういうバグでした・こう直しました」と答えられる。記録に無いことは正直に不明と言う。

### 2.5 考え中の経過共有

本応答（GPT 5.5）に時間がかかるとき（2026-07-07 戸田要望）:

- 受信直後に「どう受け取り・どう動くつもりか」の方向性の一言を GPT 5.5 で並行生成し、**30秒たっても本応答が出ていなければ先に投稿**。
- 以降**60秒ごと**に経過を上限2回＝2分超の沈黙を作らない。
- 速く返せた時は何も出さない（二重投稿で会話を汚さない）。
- 対象は必ず返事が来る経路（確認ターン・明示 @メンション）のみ。メンション無しのトップレベル投稿は分類の結果「静観」がありうるため対象外。

### 2.6 自己訂正（retract）

Chiaki AI の直前のアクション（修正依頼・リマインド等）が「誤り・宛先違い・不要」と指摘されたら: ①率直に謝って言い直す ②出した依頼に取り消し注記を編集で追記（削除しない） ③紐づく裁定を retracted でクローズ。Issue の提案に逃げない（2026-07-07）。

## 3. Codex 連携（Slack からのコード修正）

### 3.1 起動

戸田さんのコード変更の issue（バグ・変更・新機能）は、承認と同時に**既定で** Codex の作業キューへ入りランナーが即起動（cron */10 がバックストップ）。「起票だけで」と言われた場合とコード外の作業は起票のみ。rule は対象外（2026-07-08 戸田「その場でコードを編集していけないの？」＝旧「Codexで」の合言葉方式を廃止）。

### 3.2 実行

- VPS 内の Codex CLI（gpt-5.5・ChatGPT サブスク認証）が、リポジトリのローカルクローンの `codex/q<ts>` ブランチで実装＋自己テスト（workspace-write サンドボックス・30分上限・日次5件上限・同時実行1）。
- 会話スレッド発の依頼: 「作業を開始しました」の進捗と詳細な完了報告は**その会話スレッド**へ。レビュー待ち（変更あり）になったら**トップレベルにも一覧用の短い報告**（内容＋ブランチ＋スレッドリンク）。変更なし・失敗はスレッドのみ。

### 3.3 スレッド対話

報告スレッドでの戸田さんの返信を GPT が分類: 追加指示→同じブランチで続きを実装（件名は「継続：<今回の指示>」）／質問→文脈で回答／「反映して」→反映依頼として記録（反映はレビュー後）／雑談→自然な応答。Codex 実行中に届いた返信も完了後に必ず拾う（既読位置は完了処理で進めない）。

### 3.4 反映

本番反映は Claude Code のレビュー後のみ。VPS は GitHub への書き込み権限を持たない（ブランチは VPS 内・push とデプロイは Claude Code）。レビュー採用→merge→deploy→Issue 完了化→スレッドに記録。

## 5. 工数系

- 業務種別は「種別（対象）」の細粒度（全角かっこ）: 流し込み（コンテンツ/求人ページ/商品ページ）・修正（文言/コンテンツページ/求人ページ/固定ページ）等。実測の対象は ①業務名 ②同案件の開始/終了/進捗投稿 ③開始前の指示（案件名と同じ節のみ）の3段で自動推定。特定できないものは無理に振らない。
- 適正工数_DB（Notion）: 種別1行＝実測レンジ・中央値・回数・業務内容の具体例。単位は分の整数。適正工数（下限/上限）は戸田さんが決める正本。
- **夜間の自動反映**: compute-baselines（平日21時）が baseline 再構築後に DB 既存行の実測列（下限/上限/中央値/回数/最終更新）を反映。乖離は適正工数（下限/上限）が両方入っている行だけ判定（中央値がレンジ外で ON）。行の新規作成はしない（ページ作成は許可制）＝未登録の分類済み種別はログに候補として出すのみ。
